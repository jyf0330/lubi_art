# 项目全局规则

本文件适用于整个 Godot 项目。新增或修改功能时，必须遵守以下约定。

## 核心目标：数据、连接代码、美术表现三层分离

项目采用三层结构，依赖方向固定为：

`数据层 -> 连接/逻辑层 -> 美术表现层`

### 1. 数据层（Data）

- 所有业务数据必须由专门的数据代码统一定义、提供和管理；场景、美术预制体和动画脚本内不得散落业务数据。
- 在真实数据尚未提供时，使用集中管理的假数据（Mock Data）完成开发和预览。
- 假数据必须明确标注为 `mock`，集中放在数据层，保持与未来真实数据相同的数据结构和字段含义，方便后续直接替换数据源。
- 替换成真实数据时，原则上只改数据层或数据接入代码，不要求美术表现层重新制作或修改节点结构。

### 2. 连接/逻辑层（Controller / Adapter）

- 由专门的代码读取数据层的数据，并把数据转换成美术表现层需要的参数、状态和事件。
- 负责业务逻辑、状态切换、数据校验、资源选择和信号连接。
- 不在这一层临时绘制或硬编码复杂的美术节点；需要显示的内容应调用或实例化表现层预制体。
- 对表现层提供稳定、清晰的接口，例如 `setup(data)`、`play(state)`、`refresh(view_model)`，避免表现层了解真实数据来源。

### 3. 美术表现层（View / Art）

- 美术表现层只负责画面、排版、动画、特效、音画节奏和交互反馈，不负责保存或决定核心业务数据。
- 表现层通过连接/逻辑层传入的参数和状态工作，不直接读取网络、存档、全局业务数据或真实数据源。
- 美术资源、动画和动效要优先设计成可拆分、可复用、可独立预览的 Godot 场景预制体（`.tscn`）。

## 预制体与动效规则

- 能做成预制体的 UI、角色表现、动画、粒子、提示、弹窗和动效，优先做成独立预制体，不直接堆在大型主场景中。
- 每个预制体应尽量职责单一，拥有清晰的入口、可调参数和必要信号，可以被多个场景重复实例化。
- 动画时间、颜色、尺寸、贴图、文案、数值等常用表现参数应可在 Inspector 中调整，或通过公开方法传入；避免埋在脚本内部难以修改。
- 预制体应尽量自包含美术节点和动画，但不得内置不可替换的业务数据。
- 动效需要支持独立播放和快速替换；场景逻辑通过方法或信号触发，不依赖外部脚本直接操作预制体深层子节点。
- 主场景只负责组合预制体和编排流程，避免复制相同的节点树、动画轨道或特效逻辑。

## 开发时的默认做法

- 真实数据缺失时，不等待数据补齐：先在数据层补充结构完整的假数据，再通过连接/逻辑层驱动预制体。
- 新功能开始前先判断哪些表现可以拆为预制体；只要具有复用、独立调试或未来替换价值，就应拆分。
- 禁止让美术预制体直接依赖某一份假数据。假数据和真实数据必须通过同一个连接接口进入表现层。
- 如果现有代码把数据、逻辑和表现混在一起，修改相关功能时应逐步按三层结构拆分，避免继续增加新的耦合。

## 验收标准

完成一个功能前确认：

1. 数据是否由单独的数据代码集中管理，并可从假数据平滑替换成真实数据？
2. 数据是否经过连接/逻辑层后才进入美术表现层？
3. 美术表现是否不关心数据来自假数据、存档还是服务器？
4. 可复用的动画、动效或 UI 是否已拆成独立预制体？
5. 是否可以在不修改业务数据代码的情况下替换美术预制体，或在不修改预制体结构的情况下替换真实数据？
