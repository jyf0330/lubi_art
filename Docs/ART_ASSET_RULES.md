# 美术资源、界面与交付规范

本文件规定本项目新增或修改美术资源、UI、动效和预制体时必须遵守的制作与交付标准。
架构和目录职责以 [`PROJECT_STRUCTURE.md`](PROJECT_STRUCTURE.md) 为准，公开接口以
[`COMPONENT_API.md`](COMPONENT_API.md) 为准，迁入真实程序项目前同时阅读
[`PROGRAM_INTEGRATION.md`](PROGRAM_INTEGRATION.md)。

## 0. 适用范围、规则等级与冲突处理

本规范从发布之日起约束所有新增文件，以及本次任务实际修改、迁移或对外交付的组件。
历史资源不要求为了“看起来统一”一次性批量重命名；当历史资源被迁移、替换或成为新组件公开依赖时，
再按本规范治理，并同步修复全部 `res://` 引用。

```text
必须 / 禁止（MUST）       合入和交付门禁，不满足即不能标记完成
应该 / 优先（SHOULD）     默认执行；不执行时在交付说明中写明原因
可以（MAY）               根据组件规模和表现目标选择
```

文档发生冲突时按职责判断，不按文件新旧互相覆盖：

```text
项目边界与三层依赖       AGENTS.md
目录和文件归属           PROJECT_STRUCTURE.md
资源、布局、动效、验收   ART_ASSET_RULES.md
公开方法、信号、字段     COMPONENT_API.md
程序项目迁移             PROGRAM_INTEGRATION.md
```

任何例外都必须记录“规则、原因、影响范围、负责人、移除条件”。数据边界、正式规则不得进入
Prefab、`game` 不访问预制体深层节点这两类架构规则不接受临时例外。

## 1. 基本原则

- 本项目负责“显示什么、怎么动、怎么交互反馈”，不负责正式伤害、价格、胜负、存档和服务器规则。
- PSD、Figma 和原始绘图文件是设计与切图源；Godot `.tscn`、动态文本、交互状态和动画是运行时表现真相源。
- 可复用或需要独立调试的界面、动效、角色表现必须制作成独立 `.tscn` 预制体。
- `game` 只装配页面、管理页面生命周期和转发信号，不复制页面内部节点，也不访问预制体深层节点。
- Mock 数据只用于独立预览，必须通过 Provider / Controller 进入表现层，不得写进预制体。

## 2. 文件与节点命名

### 2.1 通用规则

- 新增或本次迁移的文件名、目录名、节点名、资源 key 和信号名使用英文 ASCII；玩家可见中文只放在文本内容或本地化资源中。
- 禁止使用空格、中文文件名以及 `新建文件夹`、`最终版2`、`aaa`、`未标题` 等临时名称进入正式目录。
- 同一组件的场景和主脚本使用同一英文词干，例如 `GoldCounterView.tscn` 与 `GoldCounterView.gd`。
- 不为单纯统一风格批量重命名历史资源。确需重命名或移动已被 `.tscn`、`.tres` 引用的资源时，优先在 Godot 编辑器中完成，并在修改后检查所有 `res://` 引用和运行时加载结果。

### 2.2 类型命名

```text
完整页面                 PascalCase + View       BattleMainView.tscn
可复用组件               PascalCase + View       GoldCounterView.tscn
交互槽位                 PascalCase + Slot       RouteOptionSlot.tscn
特效                     PascalCase + Effect     MonsterBiteEffect.tscn
连接/编排代码            PascalCase + Controller BattleMainController.gd
数据适配代码             PascalCase + Provider   GameDataProvider.gd
运行时节点               PascalCase              BeginTurnButton
方法、变量、信号         snake_case              refresh, begin_turn_requested
图片、图标、序列帧       snake_case              icon_coin.png, hit_0001.png
数据 key / screen_id      snake_case              shop_offers, artist_flow
```

语义信号描述意图，不描述内部控件：使用 `begin_turn_requested`，不使用
`button_1_clicked`。对外接口不得依赖节点在场景树中的序号。

## 3. 资源归档和唯一来源

- 功能专用图片放 `Features/<Feature>/Art/`；两个及以上 Feature 使用的资源放 `Shared/Art/`。
- 字体、Theme、Shader 和跨功能 UI 分别放 `Shared/Fonts/`、`Shared/Themes/`、`Shared/Shaders/` 和 `Shared/UI/`。
- 原始 PSD、Figma 导出说明或高分辨率工作源不得被运行时场景直接引用。默认保存在团队外部源文件库；只有项目明确建立 `SourceArt/` 后才放入该目录，且 `SourceArt/` 永远不是运行时依赖。
- 一张正式图片只保留一个运行时来源；其他组件通过 `res://` 引用，不复制出多个内容相同的文件。
- 迭代废弃但暂时不能删除的内容移入 `Legacy/`，并清除正式场景引用；禁止在原目录堆叠 `_old`、`_bak`、`_final2`。
- 截图、录屏、临时导出和验证日志放 `output/`，不作为正式运行时依赖。

## 4. 图片与 Godot 导入

- UI 图片优先使用无损 PNG；不需要透明通道的整屏背景可以在肉眼对比无明显损失后使用 WebP。不要只为减小仓库体积转换格式。
- Godot 4 中纹理过滤由项目默认值、CanvasItem 的 `texture_filter` 或父节点继承共同决定：像素风必须使用 nearest，非像素风 UI 默认使用 linear。需要混用时在组件根节点明确覆盖，并在真实缩放比例检查结果。
- 需要无损边缘的按钮、面板和边框使用 NinePatchRect 或 StyleBoxTexture，明确设置 patch margin，禁止靠拉伸整张图适配尺寸。
- 图标和纹理不得保留大面积透明空白；Pivot、视觉中心和点击区域需要在预制体中明确校准。
- 序列帧必须连续编号并放在独立目录，例如 `hit_0001.png`、`hit_0002.png`。
- 修改图片后必须检查 Godot reimport 结果、透明边缘、CanvasItem 过滤继承、导入压缩伪影和运行时显存占用。
- `.import` 记录源资源的导入设置，`.godot/imported/` 是本地生成缓存；两者均不得手工编辑。是否跟踪以当前仓库 `.gitignore` 和既有版本策略为准，不在单个任务中擅自改变全项目策略。

## 5. UI 布局与适配

- 当前基准画布为 `1920×1080`，Stretch 为 `canvas_items + expand`，以 `project.godot` 为真相源。修改这些值属于项目级变更，必须同步检查所有完整页面并更新本规范。
- 完整页面优先使用 `Control`、Container、Anchor 和 Offset 表达布局；禁止仅靠大量绝对坐标适配不同窗口。
- 设计稿给出的 authored 位置、尺寸、间距和层级必须保留；适配通过外层容器、Anchor 或可配置参数完成，不重画近似布局。
- 每个完整页面至少检查 `1920×1080` 基准、`2560×1080` 宽屏、`1440×1080` 窄屏。页面若声明支持更小窗口，还必须检查其声明的最小尺寸；这些尺寸用于发现布局问题，不代表程序项目最终只支持这三种分辨率。
- 文本组件必须检查空文本、中文、英文、长文本、大数值和换行；禁止通过缩小到不可读字号解决溢出。
- 可点击区域不得小于可见按钮的有效区域；不可见装饰节点不得拦截鼠标输入。
- 层级使用明确的容器和 `z_index` 规则；弹窗、提示、拖拽影子和特效不得永久遮挡主要交互。
- Theme 中可以统一表达的字体、字号、颜色、边距、圆角和按钮状态，不在多个场景中重复硬编码。

## 6. 交互状态

所有可交互组件按适用范围提供以下状态：

```text
default / hover / pressed / focused / disabled
selected / dragging / valid_target / invalid_target
loading / empty / error
```

- 状态由公开方法、ViewModel 或语义事件驱动，不由外部代码直接修改深层节点。
- 禁用状态必须同时停止输入并给出可见反馈，不能只降低透明度但仍然触发事件。
- 拖拽组件必须能恢复原位，明确合法与非法落点反馈，并在取消、页面切换后清理临时状态。
- 键鼠焦点、Esc 返回和弹窗输入阻断规则应由页面 Controller 统一连接。

## 7. 动画和特效

- 动效必须可以通过 `play(animation_data)` 或职责明确的公开方法触发，并能独立预览。
- 动画时间、颜色、尺寸、贴图和强度等常调参数使用 Inspector 导出变量或表现参数传入。
- 关键动效明确四个时点：触发、前摇、命中/落位、结束；需要业务结算的时点通过信号或回调通知 Controller。
- 动画重复播放、快速连播、页面中途关闭时必须正确停止并清理，不残留节点、Tween、材质参数或信号连接。
- 特效不能决定伤害、命中和胜负；它只表现 Controller 已提供的结果。
- 禁止把仅为一次演示成立的长流程动画硬编码进通用预制体。

## 8. 性能基线

- 不在 `_process()` 中持续执行可由信号、Timer 或状态变化触发的节点查找、资源加载和实例化。
- 高频生成的伤害数字、弹道和命中特效应支持及时释放；达到可感知规模后再评估对象池，不预先增加无必要复杂度。
- 同一资源使用 `preload`、缓存或共享引用，禁止在每帧重复 `load()`。
- 大图、序列帧和粒子提交前检查实际显示尺寸、帧数、同时播放数量和生命周期。
- 完整页面不得出现持续增长的节点数、重复信号连接或页面切换后的孤立实例。
- 新增重型 Shader、Viewport、粒子或全屏效果时，在基准设备的真实运行窗口检查帧率和显存影响，并在交付说明中记录。
- 性能比较必须使用同一设备、同一 Godot 构建和同一预览流程。没有正式设备预算前，默认目标是在 `1920×1080` 下稳定 60 FPS；若本机基线本来低于 60 FPS，则以修改前后相同流程为准，平均帧时间不得无说明恶化超过 10%。
- 页面切换或特效重复播放至少执行 10 次；节点数、孤立实例数和错误日志不得随次数持续增长。仅凭编辑器静止画面不能判定性能通过。

## 9. 公开接口和数据边界

- 可交互 Prefab 至少提供 `setup(view_model)` 或 `refresh(view_model)`，并发出语义信号。
- ViewModel 只包含表现所需字段；预制体不得接收正式存档对象、服务器连接、伤害计算器或胜负判定器。
- 贴图路径、品质样式和美术专用映射由 Provider / Adapter 转换，不写回正式 Snapshot。
- 缺少必需字段时应使用明确占位或产生可定位的错误，不得悄悄创建第二套默认业务规则。
- 新增或改变公开方法、信号、ViewModel 字段时，必须同步更新 `COMPONENT_API.md` 和相关 smoke test。

## 10. 版本管理与变更边界

- 修改前先确认文件归属；工作区已有的无关改动属于其他任务，不覆盖、不清理、不顺手格式化。
- 提交时精确选择本任务文件，不使用会混入整个工作区的宽泛提交方式。
- 不提交临时截图、长录屏、编辑器日志、系统文件和重复导出资源，除非它们是明确要求的评审证据。
- 删除或替换正式资源前先确认引用范围；无法立即删除的旧内容移入 `Legacy/` 并说明替代入口。
- 修改项目级输入、窗口、渲染、Theme 或导入默认值时，必须在交付说明中列出受影响页面。

## 11. 迁移交付清单

交付一个页面或预制体时至少包含：

```text
场景入口               res://.../ComponentView.tscn
公开脚本               res://.../ComponentView.gd
依赖资源               Art / Shared 中的准确路径
公开方法和信号         setup / refresh / play / requested
ViewModel 示例          字段、类型、空值与极值
触发与动画时序         触发、命中、结束
适配范围               基准、宽屏、窄屏、最小窗口
验证结果               smoke test + 真实 Godot 窗口
已知限制               尚未接入或需要正式程序接管的部分
```

迁入真实程序项目时保留节点结构、authored 几何、资源、动画、表现脚本和公开接口；
替换 Mock、Preview Controller、演示状态与演示规则。禁止根据截图重画近似页面。

实际交付时复制并填写 [`DELIVERY_TEMPLATE.md`](DELIVERY_TEMPLATE.md)，不要只回复“已完成”。

## 12. 按变更类型选择验证

不是每次修改都需要跑完整流程，但必须覆盖其真实风险：

| 变更类型 | 必做检查 | 可见证据 |
|---|---|---|
| 单张图片、图标替换 | 引用、透明边缘、过滤、基准尺寸 | 真实场景截图 |
| 布局、字体、Theme | 基准/宽屏/窄屏、长文本、点击区域 | 三种尺寸截图或录屏 |
| 按钮、拖拽、弹窗 | 对应 smoke、输入阻断、取消与恢复 | 真实鼠标连续录屏 |
| 动画、特效、时序 | 重播、连播、中断、清理、命中时点 | 从触发到稳定终态的录屏 |
| Provider、ViewModel、公开接口 | 契约 smoke、空值/极值、文档同步 | 测试结果和真实页面 |
| 完整页面或流程迁移 | 主入口、页面生命周期、完整玩家流 | 从真实入口开始的连续录屏 |

截图只证明静态外观；需要点击、拖拽、动画时序、页面切换或清理结果时必须使用真实交互录屏。
临时 fixture 和专项预览可以帮助定位问题，但不能替代正式主入口的集成验收。

## 13. 完成定义（Definition of Done）

以下项目全部满足后，组件才可标记完成：

1. 文件位置、名称和职责符合本规范与 `PROJECT_STRUCTURE.md`。
2. 能单独实例化，并能由集中 Mock 数据通过 Controller 正常预览。
3. 外部只使用公开方法和语义信号，不访问深层节点。
4. 替换为正式 Snapshot / ViewModel 时不需要重做节点结构。
5. 空数据、极值、长文本、禁用、错误和重复触发不会破版或残留状态。
6. 图片导入、NinePatch、字体、层级、鼠标过滤和动画清理已检查。
7. 相关场景 smoke test 通过，Godot 输出中没有新增解析错误、丢失资源或运行时错误。
8. 已在真实 Godot 窗口完成可见状态、点击、拖拽和动效验证；静态截图不能替代动态交互验证。
9. `COMPONENT_API.md`、迁移说明、`DELIVERY_TEMPLATE.md` 交付记录和必要的 ViewModel 示例已经同步。
10. 验收产物放在 `output/`，正式资源没有依赖临时文件。

不适用的项目允许标记 `N/A`，但必须写明为什么不适用；不能留空后默认通过。

## 14. 一句话检查

```text
game 负责摆页面，不负责算规则。
Controller 负责接线，不负责重画美术。
Prefab 负责表现，不负责决定结果。
Mock 负责演示，不负责成为真相源。
```
